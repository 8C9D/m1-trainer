# Refactor Opportunities Report

_Generated by the repo-cleanup-autopilot skill. Branch: `chore/repo-cleanup-autopilot`._

## 1. Repository Overview

`g1m1-trainer` is a self-hosted practice tool for the Ontario G1 (car) and M1
(motorcycle) knowledge tests. It has two cooperating parts:

- **Data pipeline (root + `scripts/`, CommonJS, Playwright):** scrapers harvest
  questions from g1.ca into `data/`, then `scripts/sync-data.js` mirrors that
  into `web/public/data/` (stripping scraper-only fields), with an optional
  image-cache flow.
- **Web app (`web/`, Next.js 16 App Router, React 19, TypeScript, Tailwind 4):**
  a client-side quiz UI that fetches the published JSON, shuffles questions,
  tracks score, and persists a "missed questions" bank in `localStorage`.

The codebase is small (~2,250 lines of source across JS/TS/TSX) and notably
well-factored already: recent history shows a deliberate refactoring streak
(centralizing the data route, test-id classification, extracting sync helpers
and the test-results component, flattening practice-test metadata). Pure logic
lives in testable modules; the data-pipeline scripts share `scripts/image-cache.js`
and `scripts/sync-helpers.js`.

## 2. Current Quality Summary

**Strengths**

- Strong unit-test coverage (84 tests across 5 files) over the pure logic:
  question parsing/validation, bank storage versioning + migration, test-id
  classification, image-URL mapping, and the sync projection.
- Clean separation of concerns: client logic (`lib/questions.ts`), server-only
  counting (`lib/questions.server.ts`), small focused components, and pure,
  filesystem-free script helpers that are unit-tested from the web project.
- Good documentation: a thorough root `README.md` and `web/AGENTS.md` guidance.
- Comprehensive `.gitignore`; no tracked build artifacts (`.next`,
  `*.tsbuildinfo`, `next-env.d.ts` are all ignored).
- CI runs test + build + lint on every push/PR.

**Baseline validation (all green before any changes):**

- `cd web && npm test` → 84 passed (5 files)
- `cd web && npm run lint` → clean
- `cd web && npm run build` → success (only the known, README-documented
  multi-lockfile warning)

**Weaknesses / smells (small and mostly cosmetic)**

- Leftover `create-next-app` boilerplate: 5 unused SVGs in `web/public/` and an
  unmodified `web/README.md`.
- The root `README.md` repo-layout section predates the recent script
  refactors (`scripts/scrape.js`, `scripts/sync-helpers.js` are not listed).
- The two data-pipeline scripts enumerate the `data/` directory with similar
  but intentionally divergent logic (not safely unifiable without a behavior
  change — see §6).

## 3. Highest-Value Refactor Opportunities

### Opportunity A — Remove unused create-next-app boilerplate SVGs

- **Location/files:** `web/public/file.svg`, `globe.svg`, `next.svg`,
  `vercel.svg`, `window.svg`
- **Problem:** Default Next.js starter assets, tracked in git, referenced
  nowhere in the repo (verified by a full-tree grep excluding
  `node_modules`/`.next`/`.git`).
- **Why it matters:** Dead assets are noise; they imply a relationship to the
  app that doesn't exist and bloat the published `public/` directory.
- **Suggested refactor:** Delete the five files.
- **Risk level:** Low (files served statically from `public/`; not imported, so
  build/test/lint cannot depend on them).
- **Expected benefit:** Less dead weight; a `public/` that contains only real,
  used assets.
- **Suggested validation:** `npm run build`, `npm test`, `npm run lint` in `web/`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

### Opportunity B — Replace boilerplate `web/README.md`

- **Location/files:** `web/README.md`
- **Problem:** Unmodified `create-next-app` README (mentions yarn/pnpm/bun,
  Vercel deploy, "edit `app/page.tsx`"). It duplicates/contradicts the real,
  comprehensive root `README.md`, which already documents web dev/build/start.
- **Why it matters:** Stale, generic docs mislead contributors and bury the
  project-specific instructions.
- **Suggested refactor:** Replace with a short, accurate stub that points to the
  root `README.md` and lists the web-specific npm scripts.
- **Risk level:** Low (documentation only).
- **Expected benefit:** Accurate, non-duplicative guidance.
- **Suggested validation:** Manual read; links resolve.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

### Opportunity C — Fix root `README.md` repo-layout accuracy

- **Location/files:** `README.md` ("Repo layout" + "Data flow" sections)
- **Problem:** The layout tree predates recent refactors: it omits
  `scripts/scrape.js` (the shared Playwright engine both root scrapers now
  delegate to) and `scripts/sync-helpers.js` (pure project/collect helpers). It
  still describes `scrape-g1-practice.js` / `scrape-m1-practice.js` as the
  scrapers, when they are now thin config wrappers.
- **Why it matters:** A repo map that doesn't match the tree erodes trust in the
  docs and makes onboarding harder.
- **Suggested refactor:** Update the layout tree and the one-line descriptions to
  reflect the current structure.
- **Risk level:** Low (documentation only).
- **Expected benefit:** Docs that match reality.
- **Suggested validation:** Cross-check every listed path against the working tree.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented

## 6. Things Not Worth Refactoring Yet

- **Unify `data/` traversal between `scripts/sync-data.js` and
  `scripts/cache-images.js`.** Both walk `data/` (top-level JSON + each
  subdir's `questions.json`), but `sync-data.js` uses a hardcoded allow-list for
  the two combined files while `cache-images.js` discovers all top-level
  `*.json`. Unifying would (a) change `sync-data.js` behavior to mirror *any*
  top-level JSON, and (b) touch filesystem-walking code that has **no** unit
  coverage. Fails the "preserve behavior" + "sufficient validation" bars —
  **skip**.
- **`next.config.ts` placeholder comment** (`/* config options here */`) —
  harmless boilerplate; not worth a commit.
- **Root `package.json` empty `description`/`author`/`keywords` and default
  `test` script** — these are the maintainer's to fill; changing the `test`
  script would be a behavior change (root has no test runner).
- **Tailwind class-string duplication** (the "All tests" link appears in
  `TestResults.tsx` and twice in `app/test/[testId]/page.tsx`) — extracting a
  shared component/constant for ~3 short call sites risks over-abstraction for
  little gain in an app this size.
- **Duplicated `ROOT`/`SRC_DIR` path constants** across the two scripts —
  centralizing 2 lines into a new module adds indirection for 2 consumers.
